# Introduction

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

The OpenClaw operational agent (`openclaw-ops.agent.md`), its supporting skills (`openclaw-cli`, `openclaw-config`, `openclaw-troubleshoot`), the operational runbook, and the `openclaw-connect.sh` script all carry Azure Container Apps (ACA) assumptions as their primary operational model. The AKS migration plans (`feature-aks-migration-1.md` and subplans SUB-001 through SUB-004) replace ACA with AKS + ArgoCD + Helm + Workload Identity as the runtime. These files must be brought in line with that new approach so the agent and skills guide operators correctly once AKS is live (and do not direct users to issue `az containerapp` commands against a decommissioned resource).

> **Timing gate:** The agent and skill content changes in this plan describe the post-migration target state. Execute Phases 1–4 as the AKS migration subplans complete (after SUB-003 is validated per environment). Execute Phase 5 (`openclaw-connect.sh`) once AKS DNS hostnames are resolvable. Phase 6 (ops runbook) can begin immediately as an additive change.

## 1. Requirements & Constraints

- **REQ-001**: All ACA-specific operational instructions (`az containerapp exec`, `az containerapp logs`, Container App revision restart, Container App FQDN derivation) must be replaced with their AKS equivalents (`kubectl exec`, `kubectl logs`, ArgoCD sync / pod rollout, static DNS hostname).
- **REQ-002**: The `openclaw` CLI workflow (loading remote credentials via `openclaw-connect.sh`, `openclaw status`, `openclaw doctor`) is unchanged and must be preserved verbatim.
- **REQ-003**: Exec fallback in the `openclaw-cli` skill must reference `kubectl exec -n openclaw deployment/openclaw` (or individual pod), not `az containerapp exec`.
- **REQ-004**: Log fallback must reference `kubectl logs -n openclaw deployment/openclaw --tail=100 --follow` (not `az containerapp logs show`).
- **REQ-005**: Config inspection and write operations that previously used `az containerapp exec --command "node /app/openclaw.mjs config get|set"` must use `kubectl exec -n openclaw deployment/openclaw -- node /app/openclaw.mjs config get|set`.
- **REQ-006**: Config seeding workflow replaces `seed-openclaw-config.sh` (ACA exec + PTY) with `seed-openclaw-aks.sh` (kubectl exec equivalent, to be created as part of SUB-003) or Helm values update + ArgoCD sync.
- **REQ-007**: Token rotation must reference a Kubernetes pod rollout (`kubectl rollout restart deployment/openclaw -n openclaw`) instead of `az containerapp revision restart`.
- **REQ-008**: The `openclaw-connect.sh` FQDN derivation must use the static DNS hostname (`paa-<env>.acmeadventure.ca`) as the primary source instead of `terraform output container_app_fqdn` or `az containerapp show`.
- **REQ-009**: The `openclaw-troubleshoot` skill must replace its ACA revision/replica diagnostics with `kubectl get pods`, `kubectl describe pod`, and `kubectl events` equivalents; the `diagnose-containerapp.sh` script reference must be replaced with a `diagnose-aks.sh` equivalent (script to be created separately).
- **REQ-010**: The ops runbook (`docs/openclaw-containerapp-operations.md`) must gain a new "AKS Operations" section covering bootstrap (kubeconfig, ArgoCD access), token rotation, config updates, log access, and pod restart — per the step noted in `feature-aks-platform-1.md` TASK-013.
- **REQ-011**: The existing ACA content in the ops runbook must be preserved and clearly marked as "Legacy (ACA)" until ACA decommission is complete per SUB-004. Do not delete ACA content in this plan.
- **REQ-012**: No secrets, resource identifiers, tenant names, or subscription IDs may appear in any updated file.
- **SEC-001**: The exec approach for `kubectl exec` must use the deployment reference (`deployment/openclaw`), not a hardcoded pod name, to avoid assuming pod identity — pod names change on restart.
- **CON-001**: The `openclaw-troubleshoot` skill references `diagnose-containerapp.sh`, which is an ACA-specific script. A `diagnose-aks.sh` replacement is out of scope for this plan; the skill should note the script is pending and provide manual `kubectl` equivalents in the interim.
- **CON-002**: The `openclaw-ops.agent.md` agent tool list references `azure-mcp-server/containerapps` for container replica state and exec fallback. Under AKS this becomes `kubectl` via the `execute` tool. The `containerapps` tool entry should remain for the ACA decommission window but be deprioritized.
- **CON-003**: Changes to `openclaw-connect.sh` must remain backward-compatible with the dev environment until ACA is fully decommissioned — the script must attempt the AKS DNS hostname first and fall back to the ACA FQDN derivation for environments not yet migrated.

## 3. Alternatives

- **ALT-001**: Delete all ACA content from skills and agent instead of preserving it with legacy markers — rejected; ACA decommission (SUB-004) has not started and ACA remains live until AKS validation is confirmed. Removing ACA content before the migration completes would leave operators without a valid fallback guide during the transition window.
- **ALT-002**: Create entirely new `openclaw-aks-cli` and `openclaw-aks-ops` skills instead of modifying existing ones — rejected; the underlying OpenClaw CLI commands are unchanged; only the infra-level fallback mechanisms differ. New skills would require updating trigger descriptions and agent wiring unnecessarily.
- **ALT-003**: Gate all changes on AKS being live before making any edits — rejected; Phase 6 (ops runbook extension) is purely additive and can be done immediately; the agent/skill updates can be drafted and reviewed now against the known AKS design without requiring a live cluster.

## 7. Risks & Assumptions

- **RISK-001**: If the ArgoCD application name or OpenClaw deployment name differs from `openclaw-<env>` / `openclaw` at implementation time, all kubectl commands referencing those names will fail silently or target the wrong resource. Mitigation: verify names with `kubectl get deployments -n openclaw` and `kubectl get applications -n argocd` before publishing the updated docs.
- **RISK-002**: `openclaw-troubleshoot/SKILL.md` references `diagnose-containerapp.sh` which is ACA-specific. The replacement `diagnose-aks.sh` is not being created in this plan. If operators invoke the troubleshoot skill during a pod crash and the script is referenced without a clear "this script is deprecated", they may attempt to run it against a non-existent Container App. Mitigation: TASK-013 explicitly notes the script is legacy/pending replacement and provides manual kubectl commands.
- **RISK-003**: The `openclaw-connect.sh` DNS-first logic added in TASK-017 uses DNS resolution as the migration signal. If a DNS change is made but AKS is not yet serving traffic (e.g., cert not ready), the script will set `OPENCLAW_GATEWAY_URL` to the AKS hostname and the CLI will fail to connect. Mitigation: operators must confirm gateway health via `openclaw status` immediately after sourcing credentials; reconnect-troubleshoot guidance is already in the skill.
- **ASSUMPTION-001**: OpenClaw Kubernetes deployment uses namespace `openclaw` and deployment name `openclaw` — consistent with the umbrella chart name and ArgoCD Application naming in `feature-aks-application-1.md`.
- **ASSUMPTION-002**: ArgoCD Application names follow the pattern `openclaw-dev` and `openclaw-prod` — consistent with the umbrella chart directories in `workloads/dev/` and `workloads/prod/`.
- **ASSUMPTION-003**: `kubectl` access is available in the devcontainer to operators running these commands; `az aks get-credentials` has been run for the target environment.
